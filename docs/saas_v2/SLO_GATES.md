# V2 SLO Gates (Phase 6B)

## Release gate metrics
These gates are evaluated from the latest load report JSON generated by `load:light` / `load:full`.

| Gate | Metric source | Threshold | Result |
|---|---|---|---|
| Gateway ACK latency | `metrics.ack_ms.p(95)` | `< 200ms` | hard fail |
| Queue start lag | `metrics.queue_lag_ms.p(95)` | `< 2000ms` | hard fail when metric present |
| Error rate | `metrics.http_req_failed.rate` | `< 0.01` | hard fail |
| Duplicate side effects | `metrics.duplicate_side_effect_rate` | `= 0` | hard fail when metric present |

## Regression tolerance
Small variance is allowed via `PERF_GATE_VARIANCE` (default `0.10` = 10%).

Effective gate example:
- ACK p95 gate becomes `200 * (1 + variance)`.

## Local run
```bash
npm run load:light
npm run perf:gate
```

## CI run
CI executes:
1. `npm run load:light`
2. `npm run perf:gate`

If gate fails, workflow fails and blocks merge (default `PERF_GATE_WARN_ONLY=false` in CI).

## Warn-only mode (rollback switch)
For temporary stabilization windows:

```bash
PERF_GATE_WARN_ONLY=true npm run perf:gate
```

- Prints failures but exits `0`.
- Default is blocking (`false`) for `main`.

## Autoscaling guidance
- If `queue_lag_p95 > 2000ms` for 5 minutes: scale worker replicas by **+30%** (minimum +1).
- If `pending_notifications` backlog exceeds configured cap for 3 consecutive checks: page on-call and investigate notifier throughput.
- If KiotViet `429` error share exceeds 5% for 10 minutes: reduce worker concurrency by 20% and increase backoff.
